# Distributed Tracing

> Source pattern: [Distributed Tracing — microservices.io](https://microservices.io/patterns/observability/distributed-tracing.html)

---

## Overview

In a microservice architecture, a single user-facing operation may touch a dozen services. Each service logs its own events, but those log lines are interleaved with thousands of other requests. Without a shared identifier, you cannot reconstruct what happened during a specific user's interaction.

Distributed tracing solves this by assigning a **Trace ID** at the entry point of every external request and propagating it through every downstream service call, so that all log entries, spans, and metrics for that request can be correlated with a single query.

---

## Core Concepts

### Trace

A trace represents the complete lifecycle of one external request — from the user's action through every service it touches. A trace is identified by a globally unique **Trace ID**.

### Span

A span represents a single unit of work within a trace:

- An HTTP handler processing a request
- A database query
- A call to a downstream service
- A message publication to Kafka

Spans form a tree. Each span has:

- A `span_id` (unique within the trace)
- A `parent_span_id` (except for the root span)
- A start time and end time (enabling latency analysis)
- A set of attributes (key-value metadata)

### Trace Context Propagation

When Service A calls Service B, it must pass the current trace context as HTTP headers, so that Service B's spans become children of Service A's span — rather than starting a new, disconnected trace.

OBSERVA4J uses the **W3C TraceContext** standard:

```http
traceparent: 00-7d2c8e4f1a3b9c2d-a3f9c2d17b44-01
tracestate:  (optional vendor-specific state)
```

The `traceparent` header encodes: `version-trace_id-parent_span_id-flags`.

## The Problem Solved

### Without Distributed Tracing

```text
14:32:01.001  [order-service]       ERROR  "Payment failed for order ORD-9912"
14:32:01.003  [payment-service]     ERROR  "Card declined: USR-445"
14:32:01.240  [notification-service] INFO  "Email sent to unknown user"
```

These three log lines are from the same user request. Without a shared identifier, there is no way to know that automatically.

### With Distributed Tracing

```text
14:32:01.001  [order-service]        ERROR  trace_id=7d2c  "Payment failed for order ORD-9912"
14:32:01.003  [payment-service]      ERROR  trace_id=7d2c  "Card declined: USR-445"
14:32:01.240  [notification-service]  INFO  trace_id=7d2c  "Email sent to user USR-445"
```

A single query for `trace_id = 7d2c` across all services reconstructs the full story.

## Integration with Quarkus

Quarkus provides native OpenTelemetry integration:

```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-opentelemetry</artifactId>
</dependency>

```

```properties

quarkus.otel.traces.sampler=always_on
quarkus.otel.exporter.otlp.traces.endpoint=${OTEL_EXPORTER_ENDPOINT:http://localhost:4317}

```

With this dependency, Quarkus automatically:

- Instruments incoming HTTP requests (creates the root span)
- Instruments outgoing HTTP calls (propagates the trace context)
- Instruments database queries (creates child spans)

OBSERVA4J adds the current `trace_id` and `span_id` to every log event, so that logs and traces are correlated without additional developer effort.

## Extracting the Real Trace ID

A common mistake is generating a UUID manually as a `traceId`. This creates a fake identifier that has no relationship to the actual distributed trace visible in Jaeger or Zipkin.

Always extract the Trace ID from the active OpenTelemetry span:

```java
public class TraceContextExtractor {

    public static String currentTraceId() {
        SpanContext context = Span.current().getSpanContext();
        return context.isValid() ? context.getTraceId() : null;
    }

    public static String currentSpanId() {
        SpanContext context = Span.current().getSpanContext();
        return context.isValid() ? context.getSpanId() : null;
    }
}
```

This integrates with real tracing infrastructure and works across microservices.

## Request ID vs. Trace ID

These are complementary, not competing:

| Identifier | Scope | Generated by | Purpose |
| --- | --- | --- | --- |
| `request_id` | Single HTTP request on a single service | OBSERVA4J JAX-RS filter | Isolate all logs for one request within one service |
| `trace_id` | Entire end-to-end user operation, across all services | OpenTelemetry SDK | Correlate logs and spans across service boundaries |

In practice, both should be present in every log event. `request_id` is useful for single-service debugging; `trace_id` is essential for cross-service diagnosis.

## Span Data

Each span records:

| Field | Description |
| --- | --- |
| `trace_id` | The root trace identifier |
| `span_id` | This span's unique identifier |
| `parent_span_id` | The parent span (null for the root) |
| `operation_name` | What this span represents (e.g., `POST /orders`) |
| `start_time` | UTC timestamp with nanosecond precision |
| `end_time` | UTC timestamp with nanosecond precision |
| `duration_ms` | Derived from start and end |
| `status` | `OK`, `ERROR`, or `UNSET` |
| `attributes` | Key-value metadata (HTTP method, DB query, etc.) |

OBSERVA4J supports exporting span data to any OTLP-compatible backend via the
OTel Collector or directly. Common targets:

| Backend | Protocol | Use Case |
| --- | --- | --- |
| **Jaeger** | OTLP (gRPC or HTTP) | Open-source trace visualisation |
| **Grafana Tempo** | OTLP (gRPC) | Grafana-native trace backend |
| **Elastic APM** | OTLP via OTel Collector | ELK stack trace visualisation |
| **Azure Application Insights** | OTLP via `quarkus-opentelemetry-exporter-azure` | Unified Azure backend (logs + traces + metrics) |
| **Zipkin** | HTTP JSON via Collector | Lightweight trace visualisation |

Configuration (Quarkus):
```properties
quarkus.otel.exporter.otlp.traces.endpoint=http://otel-collector:4317
quarkus.otel.traces.sampler=parentbased_always_on
```

## Sampling

Recording every span in a high-traffic system requires substantial storage and processing infrastructure. Sampling reduces this cost.

### Head-Based Sampling

The decision to record a trace is made at the **start** of the trace, before any spans are collected. Simple and low-overhead, but cannot selectively retain traces with errors.

### Tail-Based Sampling

The decision is made **after** the trace completes. Allows retention of all error traces and latency outliers while sampling normal traffic. Requires a buffering component (e.g., OpenTelemetry Collector with tail sampling processor).

---

Tail-based sampling delegated entirely to the OpenTelemetry
Collector. The application SDK is always configured as `always_on` — it sends
100% of spans via gRPC to a local Collector sidecar. The Collector holds a
buffer (`decision_wait`, recommended starting point: 10s, tunable per
environment) and applies the following policy:

| Environment | Sampler (App) | Collector policy |
| --- | --- | --- |
| Local / Dev | `always_on` | 100% — no discard |
| Staging | `always_on` | 50% success / 100% errors |
| Production | `always_on` | 1–5% success / 100% errors |

**Error rule (non-negotiable):** Any trace containing a span with
`StatusCode.ERROR` bypasses the probabilistic rate and is always forwarded
to storage. Applications must set `span.setStatus(StatusCode.ERROR, reason)`
in all exception handlers to activate this rule.

**Log correlation guarantee:** `trace_id` and `span_id` are injected into MDC
on every request. Even when a success trace is discarded by the Collector,
application logs retain the reference, enabling future correlation.

**Rationale:** Keeps the application free of sampling logic. Centralises the
decision in infrastructure. Guarantees zero loss of error traces without
requiring application changes per environment.

**Open caveat:** `decision_wait` must be tuned for long-running operations
(batch jobs, file processing) where spans may outlast the default buffer
window. A fixed 10s value is insufficient for those cases.

> See [Open Questions #2](OPEN_QUESTIONS.md#2-sampling-strategy) for the full
> decision record, including rationale and the open caveat on `decision_wait`
> tuning for long-running operations.

## Benefits

- **Latency analysis.** Span durations reveal which operation in a chain is causing overall slowness.
- **Service dependency mapping.** Trace data can automatically generate service dependency graphs.
- **Error propagation visibility.** See exactly which downstream call caused an upstream failure.
- **Cross-service log correlation.** Filter all logs for a user's session with a single `trace_id` query.

---

## Limitations

Per microservices.io: aggregating and storing trace data requires significant infrastructure investment. This is particularly true for tail-based sampling, which requires a stateful collector. Evaluate storage requirements before enabling 100% sampling in production.
